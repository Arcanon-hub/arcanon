## Cross-Phase Integration

**Score: 11/12 integration points wired correctly**

### Verified Integration Chains

1. **Logger → Log API** (Phase 23 → 24): `createLogger` writes structured JSON to `worker.log`; `/api/logs` reads it with component/since filtering. `dataDir` properly threaded through `createHttpServer`.

2. **Log API → Log UI** (Phase 24 → 25): `initLogTerminal()` polls `/api/logs` with `?since=` watermark and `?component=` filter. Called once in `init()` after `loadProject()`.

3. **Canvas → Project Switcher** (Phase 22 → 26): HiDPI `resize()` in `init()` scope, unaffected by project switches. `fitToScreen` button re-wired via `cloneNode` on each `loadProject()`.

4. **Interactions Teardown Cycle** (Phase 26 → 22): `project-switcher.js` calls `teardownInteractions(canvas)` before `loadProject()` which calls `setupInteractions(canvas)`. Named handlers ensure clean removal.

5. **Migration Chain** (Phase 27 → 28): Migrations 004, 005 (and unplanned 006) auto-discovered and applied in version order. No FK ordering conflicts.

6. **Upsert → Bracket** (Phase 27 → 28): `persistFindings` accepts `scanVersionId` and stamps all upserted rows. `endScan` deletes stale rows.

7. **Bracket → Cross-Project** (Phase 28 → 29): `pool.js` opens DBs and runs all migrations. `resolveDb` dispatches by path/hash/name format.

### Integration Gap

**setupControls listener accumulation** (minor, PROJ-01)
- `loadProject()` calls `setupControls()` on every invocation
- `setupControls()` adds `input`/`change` listeners to `#search` and `[data-protocol]` elements without removing old ones
- After N project switches, N sets of filter listeners fire per event
- No teardown counterpart exists (unlike `teardownInteractions`)
